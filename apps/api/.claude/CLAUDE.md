# API — Claude Guide

Implementation rules for `apps/api`. For monorepo-wide policy see the root `.claude/CLAUDE.md`.

## Stack

- NestJS (latest), TypeScript
- Prisma + PostgreSQL
- class-validator + class-transformer for DTO validation
- Swagger at `/docs` (non-production only)
- Helmet, compression, CORS configured at bootstrap
- Routes under `/v1`

## Layout

```
prisma/
  schema.prisma              Prisma data model (lives at apps/api/prisma/, not under src/)
  migrations/                generated by `npx prisma migrate dev`
src/
  main.ts                    bootstrap, global pipes, CORS, Swagger
  app.module.ts              feature module registration
  config/
    configuration.ts         typed config factory (port, db, cors)
  common/
    dto/
      pagination.dto.ts      shared PaginationDto with @Min/@Max
      paginated-response.dto.ts  paginate() helper + types from @repo/types
  prisma/
    prisma.module.ts         @Global() module
    prisma.service.ts        connect/disconnect lifecycle
  health/
    health.controller.ts     GET /v1/health
  items/                     example feature module
    items.module.ts
    items.controller.ts
    items.service.ts
    items.service.spec.ts
    items.controller.spec.ts
    dto/
      create-item.dto.ts
      update-item.dto.ts
```

## Module pattern

Every feature module follows the same shape:

```typescript
@Module({
  controllers: [FeatureController],
  providers: [FeatureService],
})
export class FeatureModule {}
```

- Thin controllers — they parse DTOs, call the service, return what the service returns
- Business logic lives in the service
- DTOs use class-validator decorators (`@IsString`, `@IsNotEmpty`, `@MaxLength`, etc.)
- Use `!` for required DTO fields (`title!: string`) — class-validator handles initialization at runtime, TS strict mode is happy

## Prisma patterns

- `PrismaService` is global — inject it directly
- Use `findUniqueOrThrow` only when you have a custom error filter; otherwise check yourself and throw `NotFoundException` with a friendly message
- Transactions: `prisma.$transaction([...])` for multi-step writes
- Add indexes in `schema.prisma` for any field you query by — don't wait for slowness

## Pagination

Use the shared helper:

```typescript
import { paginate } from '../common/dto/paginated-response.dto'

return paginate(this.prisma.item, { orderBy: { createdAt: 'desc' } }, pagination)
```

It handles `count` + `findMany` in parallel and shapes the response per `@repo/types`.

## Implementation rules

1. Every DTO has validation decorators.
2. Controllers don't import Prisma — only services do.
3. Don't log request bodies wholesale — fields may contain PII.
4. Every new module gets registered in `app.module.ts`.
5. New shared types belong in `packages/types`, not duplicated here.
6. Validation is enforced globally via the bootstrap `ValidationPipe` (`whitelist: true`, `forbidNonWhitelisted: true`).

## Configuration

- All env vars listed in `.env.example`
- Config factory in `src/config/configuration.ts` — never read `process.env` directly from features
- `CORS_ORIGINS` is a comma-separated list; empty falls back to localhost in dev

## Testing

- Service specs mock `PrismaService` with `jest.fn()` per method
- Controller specs mock the service and assert the call signature
- Health controller spec covers both connected and disconnected DB paths
- E2E config at `test/jest-e2e.json` (add specs as features grow)

## Adding a new module

1. Add the model to `schema.prisma`, run `npx prisma migrate dev --name <change>`
2. Update `packages/types/src/index.ts` if the model exposes a new enum
3. Generate or hand-write DTOs under `<feature>/dto/`
4. Service implements the business logic, throws typed exceptions
5. Controller is a thin adapter
6. Register the module in `app.module.ts`
7. Add Swagger decorators (`@ApiTags`, `@ApiOperation`, `@ApiPropertyOptional`)
8. Spec the service and controller
