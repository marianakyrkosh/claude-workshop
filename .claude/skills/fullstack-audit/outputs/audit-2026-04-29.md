# Production Audit Report

**Project**: claude-workshop (Workshop Starter monorepo)
**Audited**: 2026-04-29
**Stack**: Next.js 16.2.4 + NestJS 11 + Prisma 6 + PostgreSQL 16

## Summary

The starter is well-organized for its stated purpose — a teaching template with one full vertical slice (Items CRUD) — but it is **not production-ready** as shipped. The biggest gaps are unsurprising given the goals of the workshop: there is no authentication, no rate limiting, no production deployment artifacts (no Dockerfile, no CDK, no environment validation), and no global error filter to translate Prisma errors into proper HTTP responses. A handful of the misconfigurations (CORS regex fallback, health check semantics, missing index on `Item.createdAt`) are more subtle and would cause issues even after the obvious gaps are filled. AWS integration is intentionally absent — this report skips that domain because the project is local-first by design.

## Architecture & Code Quality

### Critical

No issues found.

### High

- [ ] **#1 No output DTOs — API leaks Prisma entities directly** ❌ OPEN — `ItemsService` returns `prisma.item.create/find/update/delete` results unchanged, and `ItemsController` returns the service result. There is no response shape contract, so any future column added to the `Item` model (a soft-delete flag, an internal note, a tenant ID) is automatically exposed to clients with no review step. The `@repo/types` package only models the pagination envelope, not the entity. Found in: `apps/api/src/items/items.service.ts:12-36`, `apps/api/src/items/items.controller.ts:14-41`, `packages/types/src/index.ts`.

### Medium

- [ ] **#2 `paginate` helper signature uses `any` and bypasses Prisma's generated types** ❌ OPEN — The shared pagination helper accepts `delegate: { count: (args?: any) => …; findMany: (args?: any) => … }` and `args: Record<string, any>`. Every call site loses Prisma's per-model type-safety on `where`, `orderBy`, `select`, and `include`. Misspelled fields or invalid filters compile cleanly and only fail at runtime. Found in: `apps/api/src/common/dto/paginated-response.dto.ts:7-17`.

- [ ] **#3 Web duplicates the `Item` shape locally instead of importing a shared type** ❌ OPEN — `hooks/use-items.ts` declares its own `interface Item { id; title; description; createdAt; updatedAt }`, while `packages/types` only exports `PaginationMeta`/`PaginatedResponse`. When the API model changes (renamed field, new required field, type narrowing), the web client compiles fine and breaks at runtime. Pagination is shared via `@repo/types` but entity contracts are not — an inconsistent contract pattern that will cause drift as more features are added. Found in: `apps/web/hooks/use-items.ts:5-11`, `packages/types/src/index.ts`.

- [ ] **#4 `paginate` helper passes the same `where` to `count` and `findMany` via `args.where` without narrowing the type** ❌ OPEN — The helper destructures `args.where` from a `Record<string, any>` and forwards it to `count`. Misuse (e.g. accidentally passing a top-level `select` instead of `where`) silently produces inconsistent counts vs. data. Combined with finding #2, the helper is an unsafe shared utility used by every paginated endpoint. Found in: `apps/api/src/common/dto/paginated-response.dto.ts:14-17`.

### Low

- [ ] **#5 No e2e tests in the API — only mocked unit specs** ❌ OPEN — `apps/api/test/jest-e2e.json` exists but no `*.e2e-spec.ts` files. The unit specs mock `PrismaService` entirely, so the global `ValidationPipe` config (whitelist, forbidNonWhitelisted, transform), the global prefix, the CORS handler, and Prisma error mapping are never exercised by tests. Regressions in `main.ts` won't be caught by CI. Found in: `apps/api/test/`, `apps/api/src/main.ts`.

## Security

### Critical

- [ ] **#6 Entire API is unauthenticated and unauthorized** ❌ OPEN — There is no `@UseGuards()`, no JWT verification, no session check, and no Next.js middleware auth. Anyone reaching the API can create, read, update, or delete any item. The starter is intentionally no-auth for teaching, but exposing this surface to the public internet without adding an auth layer is unsafe for any non-local deployment. Found in: `apps/api/src/items/items.controller.ts:10`, `apps/web/middleware.ts`, repo-wide (no auth module).

- [ ] **#7 CORS fallback allows any localhost port even when `NODE_ENV=production`** ❌ OPEN — `main.ts` reads `cors.origins` and falls back to `[/^https?:\/\/localhost:\d+$/]` whenever the env var is empty, with no environment check. If `CORS_ORIGINS` is not set in production (a realistic deploy mistake), the API silently accepts requests from any localhost origin while still sending `credentials: true`. A misconfigured deploy becomes a CSRF/credential-leak vector instead of a hard failure. Found in: `apps/api/src/main.ts:18-26`, `apps/api/src/config/configuration.ts:6-12`.

### High

- [ ] **#8 No rate limiting on any endpoint** ❌ OPEN — `@nestjs/throttler` (or equivalent) is not in dependencies and not configured. Once auth is added, login/register/forgot-password endpoints will be brute-forceable; today, write endpoints (`POST/PATCH/DELETE /v1/items`) accept unlimited traffic. Found in: `apps/api/package.json`, `apps/api/src/main.ts`.

- [ ] **#9 No global exception filter — Prisma errors leak as HTTP 500 with internal details** ❌ OPEN — The bootstrap registers `ValidationPipe` but no global `ExceptionFilter`. Prisma's `PrismaClientKnownRequestError` (P2002 unique violation, P2025 record not found, P2003 foreign-key violation) bypasses NestJS's HttpException-only response normalization and surfaces as 500 errors with full message text. P2025 should map to 404 and P2002 to 409, and stack traces should never reach clients in production. Found in: `apps/api/src/main.ts`, repo-wide (no `*.filter.ts` files in `apps/api/src/common/`).

- [ ] **#10 No security headers configured for the Next.js app** ❌ OPEN — `next.config.ts` has an empty config object with no `headers()` function. There is no Content-Security-Policy, no `X-Frame-Options`, no `Strict-Transport-Security`, no `Referrer-Policy`, and no `Permissions-Policy`. The web app relies entirely on the default Next.js headers, which are minimal. Found in: `apps/web/next.config.ts:6-9`.

### Medium

- [ ] **#11 Helmet's `crossOriginResourcePolicy` is set to `cross-origin`, weakening the default** ❌ OPEN — `app.use(helmet({ crossOriginResourcePolicy: { policy: 'cross-origin' } }))` overrides Helmet's default of `same-origin`. This is the right setting only when the API serves resources to be embedded by other origins (e.g., a public CDN). For a JSON API consumed by a single-origin frontend, the default is stricter and should be preferred. Found in: `apps/api/src/main.ts:29`.

- [ ] **#12 CORS `credentials: true` combined with the permissive localhost regex** ❌ OPEN — `enableCors({ origin: corsOrigins, credentials: true, … })` allows browsers to send cookies and `Authorization` headers cross-origin. Today there are no cookies or auth, but combined with finding #7, when auth is added the regex fallback allows any port on localhost to send authenticated requests. Local malware on a developer's machine running on `localhost:3030` could exfiltrate session cookies. Found in: `apps/api/src/main.ts:21-26`.

- [ ] **#13 No `error.tsx`, `global-error.tsx`, or `not-found.tsx` in the Next.js app** ❌ OPEN — The App Router has only `app/layout.tsx` and `app/[locale]/layout.tsx`. An uncaught render error in any client component (e.g., a missing translation key, a rendering bug in a `Card`) propagates to the framework's default error UI, which exposes the React component stack in development and a generic message in production. There is no graceful fallback or way to render a friendly 404 for unknown routes. Found in: `apps/web/app/`, `apps/web/app/[locale]/`.

### Low

- [ ] **#14 CI `npm audit` only fails on `--audit-level=critical`** ❌ OPEN — High-severity vulnerabilities pass silently. By the time a vulnerability is rated "critical" (RCE, full data disclosure), upstream advisories often already include known exploits. Found in: `.github/workflows/ci.yml:78`.

## Performance

### Critical

No issues found.

### High

- [ ] **#15 No index on `Item.createdAt` despite all paginated list queries ordering by it** ❌ OPEN — `ItemsService.findAll` calls `paginate(this.prisma.item, { orderBy: { createdAt: 'desc' } }, …)`, and the Prisma schema declares no `@@index([createdAt])`. PostgreSQL will perform a sequential scan + sort on every list query. With 100 rows this is invisible; at 100k rows it dominates response time. The starter's only model is the canonical place to set the indexing pattern, and missing it teaches the wrong default. Found in: `apps/api/prisma/schema.prisma:10-16`, `apps/api/src/items/items.service.ts:17`.

### Medium

- [ ] **#16 `update` and `remove` make a redundant `findUnique` before the actual write** ❌ OPEN — Both `ItemsService.update` and `ItemsService.remove` call `await this.findOne(id)` first, then perform the update or delete — two round-trips per write. The same NotFound semantics can be expressed by catching Prisma's P2025 in a global filter and letting the single update/delete query do the existence check. With the current pattern, every PATCH and DELETE is twice as slow as it needs to be. Found in: `apps/api/src/items/items.service.ts:28-36`.

### Medium

- [ ] **#17 No path-param format validation on `:id` in `ItemsController`** ❌ OPEN — `@Param('id') id: string` accepts any non-empty string. The model uses `cuid()` IDs but malformed IDs (e.g., `'../etc/passwd'`, a 10MB string) reach Prisma which rejects them with a generic error. There is no `@IsString @IsNotEmpty @Matches(/^c[a-z0-9]{24}$/)` decorator or pipe-based validation. Combined with finding #9, malformed IDs produce inconsistent error responses. Found in: `apps/api/src/items/items.controller.ts:27-41`.

## DevOps & Configuration

### Critical

- [ ] **#18 No environment-variable validation at startup** ❌ OPEN — `src/config/configuration.ts` reads `process.env.PORT`, `DATABASE_URL`, `CORS_ORIGINS`, and `NODE_ENV` and returns them as-is. Missing `DATABASE_URL` does not crash the bootstrap — it crashes Prisma at the first query, possibly minutes into the pod's lifetime, causing a partially-healthy deployment. There is no Zod/Joi/class-validator schema enforcing required vars or types. Found in: `apps/api/src/config/configuration.ts`, `apps/api/src/main.ts`.

### High

- [ ] **#19 No Dockerfile or production container artifacts** ❌ OPEN — There is no `Dockerfile` for the API or web app, no `.dockerignore`, no production startup command beyond `node dist/main`, no health-check directives. The README points production deploys at `/devops-cicd`, but as committed there is no path from `git push` to a running production service. Found in: repo root (no `Dockerfile`, no `.dockerignore`).

- [ ] **#20 No structured logging — only NestJS's default Logger** ❌ OPEN — `main.ts` uses `new Logger('Bootstrap')` and the rest of the codebase has no logger usage at all. There are no JSON-formatted logs, no request IDs, no correlation across services, and no log shipping. In production this means investigating an incident requires `kubectl logs | grep`, with no way to follow a request from web to API to database. Found in: `apps/api/src/main.ts:9`, repo-wide.

- [ ] **#21 Health check returns HTTP 200 even when the database is disconnected** ❌ OPEN — `HealthController.check` always returns a 200 response, with the body indicating `status: 'degraded'` and `database: 'disconnected'`. Load balancers, container orchestrators, and uptime probes evaluate HTTP status codes, not response body. A pod with a broken DB connection will continue receiving traffic until manually killed. There is also no timeout on the `SELECT 1` probe, so a hung DB makes the health check hang. Found in: `apps/api/src/health/health.controller.ts:11-24`.

### Medium

- [ ] **#22 No `apps/web/.env.example` documenting `NEXT_PUBLIC_API_URL`** ❌ OPEN — `apps/web/config/env.ts` reads `process.env.NEXT_PUBLIC_API_URL` with a localhost default, but there is no `apps/web/.env.example` listing the variable. New developers don't know it exists until they need to point the web app at a non-default API. Found in: `apps/web/config/env.ts:2`, `apps/web/` (no `.env.example`).

- [ ] **#23 Husky configured but no actual git hooks committed** ❌ OPEN — `package.json` declares `"prepare": "husky install"` and `.husky/_/` exists, but no `pre-commit`, `commit-msg`, or `pre-push` hook scripts are present. The root README and CLAUDE.md state "Lint and format before every commit", but the repo enforces nothing locally — quality only kicks in at CI time. Found in: `package.json:9`, `.husky/`.

- [ ] **#24 Root `package.json` workspaces glob `apps/*` includes the Flutter app, which has no `package.json`** ❌ OPEN — `apps/mobile/` contains no `package.json`. npm workspaces silently ignores the directory but emits warnings on some npm versions, and tools that walk the workspace tree (Turbo, ESLint, third-party CLIs) may produce confusing output. The README explicitly notes mobile is not a workspace, but the glob says otherwise. Found in: `package.json:13-15`.

### Low

- [ ] **#25 Root `prepare` script runs `husky install`, which is the deprecated v8 command on husky v9** ❌ OPEN — `husky` v9.0.0 changed the bootstrap from `husky install` to just `husky`. The current command still works but emits a deprecation notice and will break on the next major husky upgrade. Found in: `package.json:10`, `package.json:22`.

## Next Steps

Top priorities to move toward a deployable starter, ordered by risk + effort:

1. **Add a global exception filter** that maps Prisma error codes (P2025 → 404, P2002 → 409, P2003 → 409) to proper HTTP responses and strips internal details from production responses (#9).
2. **Gate the CORS regex fallback on `isDevelopment`** so production deploys with an empty `CORS_ORIGINS` fail loudly instead of opening a permissive localhost regex with `credentials: true` (#7, #12).
3. **Validate environment variables at boot** with a schema (Zod or class-validator) so missing/invalid `DATABASE_URL`, `PORT`, `CORS_ORIGINS` crash the process during bootstrap, not on first request (#18).
4. **Make the health check enforce DB connectivity in the HTTP status code** (return 503 when disconnected) and put a timeout around the probe query (#21).
5. **Add output DTOs** for the Items module so the API contract is explicit and adding internal columns to `Item` doesn't auto-leak them; export the entity shape from `@repo/types` so the web client stops redeclaring it (#1, #3).
6. **Index `Item.createdAt`** before more list endpoints copy the same `orderBy: { createdAt: 'desc' }` pattern without indexes (#15).
7. **Add a Dockerfile + `.dockerignore` (or commit the `/devops-cicd` output)** so there is a real path from commit to production (#19).
8. **Configure security headers via `next.config.ts` `headers()`** (CSP, HSTS, X-Frame-Options, Referrer-Policy) and tighten Helmet's `crossOriginResourcePolicy` back to default on the API (#10, #11).
9. **Wire `@nestjs/throttler` globally** so write endpoints (and any future auth endpoints) have a baseline rate limit before any of them are exposed publicly (#8).
10. **Adopt a structured logger** (Pino or NestJS `Logger` with JSON transport) and propagate a request ID through the request lifecycle so production incidents are debuggable (#20).
